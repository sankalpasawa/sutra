## Intent

One-command install for external user X. Claude Code plugin is a thin discovery bridge; npm package (`sutra-os`) is the real OS payload that deploys into the user's project folder. No recreation of governance logic.

## User experience

```
Day 0 — install
  $ claude plugin install sutra@sankalpasawa/sutra-plugin
  ✓ Plugin installed.

Day 0 — first use
  $ cd ~/new-project
  $ claude
  > /sutra
  
  🧭 Sutra not deployed to this folder yet. Install?
  (creates CLAUDE.md, TODO.md, os/, .claude/hooks/sutra/)
  > yes
  
  [runs: npx github:sankalpasawa/sutra-os init]
  ✓ 24 hooks installed
  ✓ 18 OS docs copied
  ✓ Templates written
  ✓ Sutra is active. What do you want to do today?

Day 1..N — subsequent sessions
  $ claude
  > /sutra
  
  🧭 Sutra CEO Day — [date]
  [dashboard renders from os/STATUS.md, OKRs.md, TODO.md]
```

Plugin has **zero governance logic**. Plugin "recreates" nothing.

## Subagent propagation (the deployment insight)

After install, hooks live in the user's folder at `.claude/hooks/sutra/`. These hooks fire for ANY Claude Code process working in that directory — parent session OR subagents spawned via Task tool. The deployment model is folder-scoped, not plugin-scoped.

This means:
- Plugin only matters for first-run install + CEO day UI.
- After install, the plugin is effectively optional. User X could uninstall the plugin and Sutra would keep working because the real governance lives in `.claude/hooks/sutra/` inside their folder.
- Subagent inheritance is automatic because subagents spawn in the same working directory.

## Update flow

```
Source of truth: asawa-holding repo (private)
  ↓ hooks change (e.g., new check added to dispatcher-pretool.sh)
  ↓ copy into sutra/package/hooks/ via build step (sync script)
  ↓ commit + push sutra/package/ → sankalpasawa/sutra-os repo
  ↓ (eventually) npm publish sutra-os@X.Y.Z
User runs: npx sutra-os@latest update
  ↓ install.mjs updates their .claude/hooks/sutra/ in place
Plugin version is rarely bumped (plugin = stable thin bridge).
```

## v0.2 success criteria

1. External user runs `claude plugin install sutra@sankalpasawa/sutra-plugin` → plugin loads.
2. External user types `/sutra` in a folder → plugin detects absent install → runs `npx github:sankalpasawa/sutra-os init` → folder now has CLAUDE.md, TODO.md, os/, .claude/hooks/sutra/.
3. External user restarts Claude Code in that folder → hooks fire automatically → governance applies to every turn.
4. External user types `/sutra` again → plugin detects install present → renders CEO day dashboard from os/*.md.
5. `leak-audit.sh` passes against the plugin + the published `sutra-os` repo (no "asawa" strings).

## Risks

| Risk | Mitigation |
|---|---|
| `install.mjs` has bugs that break first install | Test locally in a throwaway folder before publishing. |
| Subagent hooks don't actually fire from `.claude/hooks/sutra/` | Verify with a test case during founder's fresh-user test. |
| Plugin's bash injection (for `npx` call) hits Claude Code permissions | Use `!`npx ...`` inline execution in the command markdown; test in plugin dev mode. |
| `sutra-os` public repo leaks Asawa strings | Run leak audit before pushing; extend to scan sutra/package/ too. |
| User runs `/sutra` but plugin can't find `npx` | Command detects + prints "install Node.js first" message. |
